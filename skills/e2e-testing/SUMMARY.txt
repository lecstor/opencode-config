SKILL OVERVIEW
================================================================================

A comprehensive, production-ready guide to writing robust end-to-end tests with
Playwright, following React Testing Library best practices.

Core Philosophy: Test user behavior, not implementation details.

KEY PRINCIPLES:
  1. Use semantic locators (getByRole > getByLabel > ... > NEVER CSS/XPath)
  2. Use web-first assertions with auto-waiting (await expect(...))
  3. Test visible user behavior, not internal state
  4. Isolate tests - each test should be completely independent
  5. Embrace Playwright's auto-waiting - no manual timeouts needed
  6. Accessibility is free - semantic locators verify it automatically

CRITICAL CONCEPTS
================================================================================

LOCATOR PRIORITY (NEVER USE CSS/XPATH):
  1. getByRole (most preferred)
  2. getByLabel
  3. getByPlaceholder
  4. getByAltText
  5. getByTitle
  6. getByTestId (use sparingly)
  7. getByText (fallback)
  8. CSS/XPath (NEVER use)

WEB-FIRST ASSERTIONS (auto-waiting built-in):
  await expect(element).toBeVisible()
  await expect(element).toHaveValue('text')
  await expect(element).toBeChecked()
  await expect(page).toHaveURL('/path')
  → Waits up to 5 seconds automatically
  → NO manual setTimeout needed
  → Clear error messages on failure

RED FLAGS (Refactor immediately):
  ❌ await page.waitForTimeout(...)
  ❌ page.locator('div.modal > form > input')
  ❌ expect(await element.isVisible()).toBe(true)
  ❌ window.__state.value
  ❌ Tests depending on other tests

BEST PRACTICES:
  ✅ Test user behavior, not implementation
  ✅ Use semantic locators exclusively
  ✅ Use web-first assertions with await
  ✅ Isolate tests completely
  ✅ Embrace Playwright's auto-waiting
  ✅ Accessibility testing is automatic

KEY TAKEAWAYS
================================================================================

1. USE SEMANTIC LOCATORS
   • getByRole, getByLabel, getByAltText verify accessibility
   • Never use CSS/XPath - they're brittle and don't verify anything
   • CSS chains break with any DOM refactoring
   • Semantic locators survive refactoring

2. USE WEB-FIRST ASSERTIONS
   • await expect() automatically retries for up to 5 seconds
   • No manual timeouts needed
   • Clear error messages when assertions fail
   • Passes as soon as condition is true (not timing-dependent)

3. TEST USER BEHAVIOR
   • Test what users see and do
   • Not internal state or props
   • Refactoring doesn't break behavior tests
   • Tests serve as user documentation

4. ISOLATE TESTS
   • Each test completely independent
   • Use beforeEach for common setup
   • Tests can run in any order
   • Tests can run in parallel

5. EMBRACE AUTO-WAITING
   • Playwright automatically waits for actionability
   • No setTimeout needed
   • No waitForSelector needed
   • Let Playwright handle the synchronization

QUICK REFERENCE
================================================================================

MOST COMMON LOCATORS:
  // Form fields
  page.getByLabel('Email')
  page.getByPlaceholder('Enter email')
  
  // Buttons
  page.getByRole('button', { name: 'Submit' })
  
  // Text
  page.getByText('Success!')
  
  // Images
  page.getByAltText('Logo')
  
  // Dialogs
  page.getByRole('dialog')
  
  // Tables/lists
  page.getByRole('row')
  page.getByRole('listitem')

MOST COMMON ASSERTIONS:
  await expect(element).toBeVisible()
  await expect(element).toHaveValue('text')
  await expect(element).toBeChecked()
  await expect(element).toBeDisabled()
  await expect(element).toContainText('text')
  await expect(element).toHaveCount(5)
  await expect(page).toHaveURL('/path')

MOST COMMON ACTIONS:
  await element.click()
  await element.fill('text')
  await element.check()
  await element.selectOption('value')

SETUP PATTERN:
   test.beforeEach(async ({ page }) => {
     // Setup runs before each test
     await loginAs(page, 'user@example.com');
   });

   test('can do something', async ({ page }) => {
     // Test code here
   });

DEBUGGING QUICK REFERENCE:
   When test fails despite code and test appearing correct:
   
   1. MANUAL REPLICATION (verify actual behavior)
      - Follow exact test steps manually in browser
      - Document what actually happens
      - Check element visibility, text, timing
   
   2. LOG ANALYSIS (examine what Playwright recorded)
      npx playwright test --debug test-name.spec.ts
      npx playwright test --trace on
      npx playwright show-trace trace.zip
   
   3. CATEGORIZE THE ISSUE
      - Playwright (locator, timing, assertion issues)
      - Code (application bugs, rendering)
      - Environment (mocking, seeding, config)
      - Timing (race conditions, delays)
      - Data (missing or incorrect test data)
   
   4. APPLY FIX (based on category)
      - See DEBUGGING-FAILING-TESTS.md for each category
   
   5. ESCALATE IF NEEDED (provide user with detailed info)
      - What the test does
      - What error/behavior is happening
      - What you've already verified
      - What category you suspect
      - Relevant logs or screenshots
