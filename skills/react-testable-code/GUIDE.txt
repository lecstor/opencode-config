KEY INSIGHT
───────────

Good React code = Testable React code

When you use:
  ✓ Semantic HTML
  ✓ Proper accessibility
  ✓ Clear component boundaries
  ✓ Explicit data flow

Then Playwright tests work automatically because they query the accessibility
tree—the same tree assistive technologies use.

CORE CONCEPTS
─────────────

Semantic HTML
  Where: SKILL.md, ACCESSIBILITY-FOR-TESTING.md, EXAMPLES.md, ANTI-PATTERNS.md
  Why: Playwright queries accessibility tree, not DOM
  Example:
    ❌ <div onClick={...} role="button">Click</div>
    ✅ <button onClick={...}>Click</button>

Accessibility = Testability
  Where: ACCESSIBILITY-FOR-TESTING.md (entire document)
  Why: Accessibility tree = testing tree
  Example:
    ✅ <button aria-label="Search">🔍</button>  // testable
    ❌ <button>🔍</button>                        // not testable

Component Boundaries
  Where: SKILL.md, COMPONENT-ARCHITECTURE.md, EXAMPLES.md
  Why: Single responsibility makes testing easy
  Example: Separate SearchForm, FilterPanel, Results components

State Placement
  Where: SKILL.md, COMPONENT-ARCHITECTURE.md, ANTI-PATTERNS.md
  Why: State must be accessible to components that need it
  Example: Filter state in parent, not in FilterPanel component

Data Flow
  Where: SKILL.md (entire Data Flow Patterns section)
  Why: Unidirectional flow makes behavior predictable
  Example: Props down (parent→child), Callbacks up (child→parent)

Composition
  Where: SKILL.md, COMPONENT-ARCHITECTURE.md, EXAMPLES.md
  Why: Composing simple components is better than configurable components
  Example: Tabs, Accordion, Modal compounds instead of huge components


TESTING PHILOSOPHY
──────────────────

Tests should verify BEHAVIOR, not IMPLEMENTATION

❌ Implementation tests (brittle):
  test('input has class "search-input"', ...)
  test('button has data-testid="submit"', ...)
  test('component state is...', ...)

✅ Behavior tests (resilient):
  test('user can search by entering text', ...)
  test('form submits on button click', ...)
  test('results appear after searching', ...)

Use getByRole, getByLabel, getByText — not CSS classes or test IDs


QUICK REFERENCE
───────────────

Find button by accessible name:
  await page.getByRole('button', { name: /submit/i }).click()

Find input by label:
  await page.getByLabel('Email Address').fill('user@example.com')

Find text on page:
  await expect(page.getByText('Success')).toBeVisible()

Find heading:
  await page.getByRole('heading', { level: 1 })

Find element with aria-expanded:
  await expect(button).toHaveAttribute('aria-expanded', 'true')

Find alert/error message:
  await expect(page.getByRole('alert')).toContainText('Error')

Wait for live region:
  await page.waitFor(() => getByText(/results/i).isVisible())
